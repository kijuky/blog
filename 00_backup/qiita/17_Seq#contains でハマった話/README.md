# 経緯

とある改修にて、あるリスト aList から、 vaList に含まれる要素を除いたリスト list を作ることになりました。ただし aList は型 A の、vaList は型 VA のリストでした。A と VA は表示用に使うクラスだったりの変換が入っていますが、概念としては同じものを指しており、それぞれに定義された id プロパティが同じ場合、同じものを指します。

list から何かを条件にして除くメソッドはそのものズバリ filter が用意されているので、それを使いました。こんな感じに：

```scala
  val list = aList.filterNot(a => vaList.map(_.id).contains(a.id))
```

これはうまく動きませんでした。なぜでしょうか。それは A#id が Int 型で、 VA#id が String 型だったからです。イメージ：

```scala
case class A (
  val id: Int,
  ...
)
```

```scala
case class VA (
  val id: String,
  ...
)
```

（表題にあるように）Seq#contains は引数に Any 型を取るので Seq[String]#contains の引数に Int 型の値を入れてもコンパイラ的にはOKだったんですね。なるほどー！

# 反省点とか愚痴とか

## テストで保護できてなかった

今回の不具合は画面に表出するくらいの不具合だったので、これが出たということは表示確認もできてなかったし、テストで保護できてなかったということです。特にこういうロジックはテストで保護すべきでした（後でテスト書きました）。

## なんで型が違うの？

そもそも A#id と VA#id でなぜ型が違うのでしょうか... そこは統一したほうが良かったかもしれませんが、今回の改修範囲からは外れてしまうため（影響範囲が広がるため）、統一は次回のリファクタまで持ち越しました。もしリファクタするならきちんと HogeId 型みたいなのを作ったほうがいいかもしれません。

## contains で異なる型が引数に指定されたら警告して欲しい

contains で型が異なる場合は絶対に false なのだから（そうとも限らないんだろうか...?）、異なる型が引数に指定された場合にはコンパイルエラーか警告して欲しいな、と思いました。だけど引数の型が Any だから難しいのかな...

# まとめ

- テスト書こう。テストが無理ならせめて１度位は表示確認しよう。
- 同じ概念を表すものは(AからVAに)ハコが変わっても、同じ型（IntならInt, StringならString）を使おう。可能ならその概念を表す型(HogeId)を作ろう。
- contains にはどんな型でも入れられるから、型パズルを過信しないようにしよう。
